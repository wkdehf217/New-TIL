# 3/12 gpt 대화를 통해 코드의 문제점 파악


1. 카프카 메시지 중복 문제 
	-> 상태 체크 or EventID 저장으로 중복 체크

+

Kafka는 at-least-once delivery 특성 때문에 메시지가 중복 소비될 수 있어
Consumer에서 Idempotent 처리를 구현했습니다.


2. Dual Write Problem 문제
	-> db저장과 kafka 메시지 발행 타이밍이 맞지 않는것. 트랜잭션 문제
	-> DB commit 성공 후 Kafka 발행 으로 처리

+ DB commit 성공
→ Kafka publish 실패 가 여전히 발생할 수 있기 떄문에 완벽한 처리 방법이 아님.
그래서 보통 사용하는 것이	Transactional Outbox Pattern

: DB와 Kafka 메시지 발행 사이의 Dual Write 문제를 방지하기 위해
Outbox 패턴을 적용하거나 Kafka 트랜잭션을 고려했습니다.


3. 카프카와 스프링의 연결 문제
	-> 내부와 외부 연결을 둘 다 신경써야 함
	-> 지금 구조 유지하려면 listener 이름을 명확히 분리해야 함

+ 실제로 중요한 것은

KAFKA_ADVERTISED_LISTENERS

예

INSIDE://kafka:29092
OUTSIDE://host:9092

즉

Docker 내부 서비스
외부 클라이언트

둘 다 연결해야 합니다.


4. db 분리 문제 (하나의 db만 사용했을 떄)
	-> MSA 는 서비스별 DB 분리가 원칙

	예시 답변 👇

현재 프로젝트에서는 인프라 단순화를 위해 PostgreSQL을 하나의 인스턴스로 운영하고 있지만, 서비스별로 스키마와 Repository 레이어를 분리하여 논리적으로 데이터 독립성을 유지했습니다.
실제 운영 환경에서는 서비스별 데이터베이스를 분리하고 Kafka 이벤트 기반으로 데이터 동기화를 할 계획입니다.

+ 실제 운영 환경에서는 서비스별 DB를 물리적으로 분리할 계획입니다.

6. 모든 통신을 kafka 로만 하면 안될까?
	-> 동기와 비동기 통신의 역할이 다름.
	-> 외부 API : REST
	    서비스 이벤트 : Kafka

단점 1. 즉시 응답 불가능
단점 2. 디버깅이 어려움
단점 3. 데이터 일관성 문제
성능 또한 동기가 좋을때도 있고 비동기가 좋을때도 있음
 
+ 외부 요청은 동기 REST, 서비스 간 이벤트는 Kafka 기반 비동기 통신으로 분리했습니다.


7. Saga 패턴에서 보상 트랜잭션을 어떻게 처리했나요?
	-> 지금 예약 하나를 진행하려면

예약 생성
좌석 예약
결제
티켓 생성
알림 발송

이렇게 여러 서비스가 필요함 
-> 그래서 saga 패턴 등장 : 여러 Local Transaction을 이벤트로 연결 + 여러 Local Transaction을 이벤트로 연결

예를 들어 결제가 실패하면? 

Seat 예약 취소
Booking 상태 FAIL

현재 이벤트 기반으로 처리 해 놓은 상태.

+ 그래서 saga 패턴 등장 을 아래로 바꿈
Saga 패턴은 여러 서비스의 Local Transaction을 이벤트 기반으로 연결하고,
중간 실패 시 보상 트랜잭션을 실행하여 데이터 일관성을 유지하는 패턴입니다.
