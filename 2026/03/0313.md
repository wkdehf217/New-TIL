## 5. 동시성 제어

- 결제 단계에서 외부 결제 API 호출 시 요청 지연이나 외부 시스템 장애로 인해 병목이 발생할 수 있음 
또한 여러 사용자가 동시에 같은 좌석을 예약할 수 있기 때문에 동시성 제어가 필요함

- 이를 위해 **비관적 락과 분산 락을 함께 적용**
<br/>

>  - 분산 락 (Distributed Lock)

    - 여러 서버 인스턴스에서 동일한 자원에 접근할 때 발생하는 동시성 문제를 해결하기 위해 사용

    - Redis 등을 이용하여 분산 환경에서 락을 관리
    - 여러 서버가 동시에 같은 자원을 처리하지 못하도록 제어

    - 이를 통해 **MSA 환경에서도 데이터 일관성을 유지**할 수 있다
      
>  - 비관적 락 (Pessimistic Lock)

    - 데이터를 조회하는 시점부터 DB 레벨에서 락을 걸어 다른 트랜잭션의 접근을 제한하는 방식

    - 조회 시 `SELECT ... FOR UPDATE` 등을 사용하여 row lock 획득
    - 다른 트랜잭션은 해당 락이 해제될 때까지 대기

    - 이를 통해 **동시에 같은 좌석을 예약하는 상황을 방지**할 수 있다

## 6. MSA

- 트래픽 급증과 장애 상황에서도 안정적으로 운영하기 위해 클라우드 기반의 인프라(AWS EC2, RDS) 와 자동화된 배포 시스템(GitHub Actions)을 결합하여 구현
- Docker 및 Docker Compose를 활용한 컨테이너화된 배포로 효율성 증대

## 8. 인증 처리

- 보안과 관리 효율성을 위해 각 서비스 대신 Gateway에서 사용자 정보를 중앙 집중적으로 검증
- MSA에 친화적이며 선언적 방식으로 간결한 코드가 장점인 FeignClient를 채택

## 9. 비동기 처리

- 결제, 예약 확정, 알림 전송과 같은 작업은 외부 API 호출 또는 추가적인 데이터 처리가 필요하기 때문에<br/>
동기 방식으로 처리할 경우 응답 지연 및 시스템 병목이 발생할 가능성이 있음

- 사용자의 요청에 대해 핵심 데이터 처리는 우선적으로 수행하고,<br/>
이후 발생하는 후속 작업(예: 이벤트 처리, 메시지 발행 등)은 Kafka를 통해 비동기로 처리하여<br/>
다음과 같은 효과를 기대할 수 있음

  - 사용자 요청에 대한 응답 속도 개선
  - 외부 시스템 장애로 인한 서비스 영향 최소화
  - 서비스 간 결합도 감소

## 10. 결제 실패 재처리 시스템

- 좌석 결제 실패 시, 사용자에게 명확하게 상태를 전달하여 편의성을 증대하기 위한 목적
  
  - 서버 리소스 낭비를 줄이기 위해 Retry Queue를 채택하고, 지연 발생을 위해 Scheduler와 결합
  - 스케줄러는 스레드 풀을 활용    
    - Kafka 메시지를 처리하며 결제 재시도를 관리
    - 재시도 작업을 비동기로 스케줄링

## 11. Kafka, Redis 서비스 성능 향상

- 기존 시스템은 동기식 처리가 주로 이루어져 대량의 데이터를 처리할 때 서버 부하가 증가하고, 성능이 저하되는 문제 발생
  
- 예를 들어, 결제나 좌석 조회 등의 작업이 서버에서 순차적으로 처리되면서 성능에 영향을 미쳤기 때문에 대기열 처리가 필요하다고 판단하여 Kafka와 Redis 도입

## 12. Kafka: 메타데이터 전송 문제

- Kafka 기반 비동기 메시지를 처리할 때 Producer와 Consumer가 서로 다른 서비스에 위치하게 됨

- 이때 서비스 간 이벤트를 전달하는 과정에서 메시지에 포함되는 메타데이터의 범위를 어떻게 설정할 것인지에 대한 문제가 발생

예를 들어,

  - 서비스 1 : 사용자로부터 데이터 `a`, `b`를 수신
    
  - 서비스 2 : 데이터 `a`만 필요
    
  - 서비스 3 : 데이터 `a`, `b` 모두 필요

이 경우 서비스 간 이벤트 전달 시  
필요한 데이터만 전송할 것인지,  
아니면 전체 데이터를 계속 전달할 것인지에 대한 설계 고민이 발생함

메시지 크기가 불필요하게 커질 경우 네트워크 비용 및 메시지 처리 부하가 증가할 수 있기 때문

> 해결 방법 검토

- 메타데이터 크기가 작고 단순한 경우  
  → 필요한 데이터만 포함하여 메시지 전송

- 데이터 추적성이 중요한 경우  
  → 전체 데이터를 이벤트에 포함하고 각 서비스는 필요한 데이터만 사용

- 메시지 크기 최적화가 필요한 대규모 시스템  
  → 외부 저장소(DB, Object Storage)에 데이터를 저장하고 참조 ID만 전송

- Kafka 기능 활용  
  → Kafka Header를 이용하여 보조 메타데이터 전달

> 최종 결정

- 본 프로젝트에서는 **데이터 추적성과 이벤트 흐름의 명확성을 중요하게 고려하여**  
이벤트 메시지에 전체 데이터를 포함하고 각 서비스에서는 필요한 데이터만 사용하도록 설계함

- 메타데이터 크기가 커지는 경우에는  
메시지에 전체 데이터를 포함하기보다는 **참조 ID 기반 이벤트 설계**로 확장할 수 있도록 고려함

## 13. Redis: 분산 락을 사용하여 동시성 문제 해결

- 한 좌석에 대해 여러 사용자가 동일한 좌석을 동시에 예약하려고 할 때, 동시성 문제로 인하여 데이터 정합성이 부정됨
  
예를 들어,<br/>

 A 사용자에게는 예약 가능한 좌석이라고 떠서 예약을 시도중인데, 그 좌석은 이미 B 사용자에게 예매가 된 상황

> 해결 방법

- Redis 분산 락 적용
  - 좌석 변경 시도 시 seatId를 사용하여 Lock Key를 설정함
  - Lock Key를 통해 좌석의 예약가능 여부를 리턴, 분산 락을 적용하여 자원 변경이 이루어지지 않도록 제한하여 동시성 문제 차단
  - Lock 만료 시간을 3분으로 설정하여 좌석 변경이 완료된 후에도 3분동안은 접근을 제한하도록 설정함

- 분산 락 구현 방법
  - SETNX 락 방식을 사용하였음
    - Redis에서 키가 존재하지 않으면 값을 설정하는 방식
    - opsForValue().setIfAbsent() 메서드를 사용하여 락을 얻고, 락을 설정하는 키가 없다면 락을 획득하고, 이미 락이 걸려 있는 경우에는 획득하지 못하는 구조
    - 키 만료 시간을 3분으로 설정
    - 간단한 명령어로 비교적 사용하기 쉬운 방식

## 14. 배포: EC2 서버 다운 이슈

- EC2 인스턴스가 메모리 부족으로 인해 자주 다운되는 문제가 발생
 
- 애플리케이션의 메모리 사용량 증가로 인해 메모리 부족 현상 발생
  
- t2.micro 인스턴스의 메모리 용량(1GB)이 애플리케이션을 운영하기에 부족
  
> 해결 방법: Kafka를 활용한 비동기 처리

- 스왑 메모리 추가

- Docker Compose에 리소스 제한 추가

- EC2 인스턴스 업그레이드
