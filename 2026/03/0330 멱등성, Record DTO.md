# 멱등성, Record를 DTO로 사용하는 이유

### 멱등성이 뭐지?
- 연산을 여러 번 적용하더라도 결과가 달라지지 않는 성질
- HTTP 메서드의 멱등성은 동일한 요청을 한번 보내는 것과 여러번 보내는 것이 서로 동일한 효과를 지니며,
  <br/> 서버의 상태도 동일하게 남을 경우에 멱등하다고 한다

### 멱등성의 활용 방법
- 어떤 이유로 인해 전송 커넥션이 끊어졌을 때, 멱등성은 클라이언트가 다시 같은 요청을 해도 되는가에 대한 판단 근거가 될 수 있다
- 멱등하다면? 요청을 재시도할 때 같은 서버의 상태를 보장하기 때문에 문제가 없음
- 멱등하지 않다면? 재시도 요청 시 중복요청을 보내 문제를 발생시킬 수 있음
  - ex) 사용자가 결제하는 시점에 타임아웃으로 인해 정상 응답을 못받는 경우
  - 멱등하지 않으면 -> 결제 API 경우에는 결제가 성공했는지 수동으로 확인하고 재요청해야 함
  - 멱등하다면 -> 안심하고 여러 번 요청할 수 있으며 중복 요청으로 발생하는 문제(중복 결제)를 방지할 수 있음

### Record를 DTO로 사용하는 이유
- Record는 무엇일까?
  - 우선 생긴 건 이렇게 생겼다
    ```
    public record User(Long id, String name) {}
    ```
    이것이 뜻하는 의미는?
    ```
    public final class User {
    private final Long id;
    private final String name;

    public User(Long id, String name) {
        this.id = id;
        this.name = name;
    }
    
    public Long id() { return id; }
    public String name() { return name; }
    }
    ```
    - "불변성" 을 기본으로 한다
    - 기존의 클래스와 달리 모든 필드가 final 키워드로 선언된다
    - 자동으로 생성자, getter, equals(), hashCode(), toString() 등 메서드를 생성함으로써 보일러 플레이트 코드를 줄일 수 있다/
    - 이러한 특성으로 멀티 스레드 환경에서 데이터가 의도치 않게 변경되고 안전하게 전달할 수 있다
    - 
  - 왜 등장했을까?

    - 기존 Java는 너무 장황했기 때문
    
      DTO 하나 만들 때:
      
      필드
      생성자
      getter
      equals/hashCode
      
      -> 반복 코드 너무 많음
      
      그래서 Java 14부터:
      
      -> “데이터 클래스는 간단하게 만들자” → record 등장

- DTO의 본질은 데이터를 전달만 하는 객체이기 때문에 Record의 개념과 완벽하게 일치함
  - ! 단, extends를 사용하여 다른 클래스를 상속할 수 없고, 필드가 final로 선언되기 때문에 확장이 어렵다
